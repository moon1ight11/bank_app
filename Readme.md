# BankApp API
Серверная часть банковского приложения на Go. 
Реализует REST API для управления пользователями, банковскими счетами и транзакциями с ролевой моделью доступа.

## Возможности
- **Аутентификация** — JWT-токены в HttpOnly cookies
- **Управление пользователями** — регистрация, обновление профиля, удаление
- **Ролевая модель** - Basic, Verificator, Admin
- **Управление банковскими счетами** — создание, просмотр, удаление
- **Транзакции** - Переводы между счетами (Basic), пополнение и списание (Verificator, Admin)
- **Административные функции** - Создание администраторов и верификаторов, изменение ролей пользователей
- **Кэширование** — использован Redis для хранения часто запрашиваемых данных
- **Логирование** — структурированное логирование с ротацией файлов
- **Контейнеризация** — Docker и Docker Compose для локальной разработки
- **Мониторинг** — использован Prometheus для наблюдения за количеством запросов, ошибками, длительностью, активными горутинами
- **CI/CD** — автоматическая сборка и публикация образа в GitHub Container Registry

## Технологии
- **Основной язык** - Go 1.25
- **HTTP-фреймворк** Gin
- **Основная база данных** - PostgreSQL
- **Миграции базы данных** - Goose
- **Аутентификация** - JWT 
- **Хэширование паролей** - bcrypt
- **Конфигурация** - Viper
- **Кэширование** - Redis (cache-aside)
- **Мониторинг** - Prometheus + Grafana
- **Контейнеризация** - Docker, GitHub Container Registry (GHCR)
- **CI/CD** - GitHub Actions (автоматическая сборка при push)
- **Архитектура** - Clean Architecture

## Особенности реализации
### Транзакции
Все операции, затрагивающие несколько таблиц, а также операции, требующие консистентности данных, используются в рамках SQL-транзакций с автоматическим откатом при ошибках.

### Кэширование
Реализована стратегия cache-aside:
- Данные сначала запрашиваются из Redis
- При кэш-промахе выполняется обращение к PostgreSQL
- Автоматическая инвалидация кэша при операциях создания, обновления или удаления

### Graceful Shutdown
При получении сигнала завершения (SIGINT, SIGTERM) сервер:
1. Прекращает принимать новые соединения
2. Ожидает завершения активных запросов
3. Корректно закрывает соединения с БД и Redis

### Логирование
Структурированное логирование с уровнями INFO, ERROR, FATAL
Запись логов в файл
Единый формат: [timestamp] LEVEL: message key=value

### Безопасность
Пароли хэшируются с помощью bcrypt (cost factor = 10)
JWT-токены хранятся в HttpOnly cookies
CORS-настройки для безопасного взаимодействия с фронтендом
Валидация всех входных данных (email, временные зоны, UUID)

### CI/CD
GitHub Actions настроен на:
1. Автоматическую сборку Docker-образа при push в репозиторий
2. Публикацию образа в GitHub Container Registry (ghcr.io)
3. Тегирование образа как latest

## Установка и запуск
### Предварительные требования
- Docker & Docker Compose
- Go 1.25+ (для локальной разработки)

### Локальный запуск с Docker
#### Создание файла конфигурации
См. файл config.example.yml
#### Запуск всех сервисов
docker-compose up -d
#### Запуск приложения
go run cmd/api/main.go

Сервер будет доступен по адресу: http://localhost:8080
Prometheus будет доступен по адресу: http://localhost:9090
Grafana будет доступен по адресу: http://localhost:3000 (логин/пароль: admin/admin)
