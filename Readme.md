## Возможности
- **Аутентификация** — `JWT`-токены в HttpOnly cookies
- **Управление пользователями** — регистрация, вход, обновление профиля, удаление
- **Ролевая модель** — Basic (переводы), Verificator (пополнение/списание), Admin (управление персоналом)
- **Управление банковскими счетами** — создание, просмотр, удаление с поддержкой мультивалютности (`RUB`, `EUR`, `USD`)
- **Транзакции** — внутренние переводы, пополнение и списание средств с ACID-гарантиями
- **Кэширование** — `Redis` (cache-aside) для хранения пользователей и счетов с автоматической инвалидацией
- **Мониторинг** — экспорт метрик в `Prometheus` и визуализация в `Grafana`
- **Логирование** — структурированное логирование с ротацией файлов
- **Контейнеризация** — `Docker` и `Docker Compose` для локальной разработки
- **CI/CD** — автоматическая сборка и публикация образа в `GitHub Container Registry`

## Технологический стек
- **Основной язык** — Go 1.25
- **HTTP-фреймворк** — Gin
- **Основная база данных** — PostgreSQL 17
- **Миграции базы данных** — Goose
- **Аутентификация** — JWT (golang-jwt v5)
- **Безопасность:** bcrypt для хэширования паролей (cost factor = 10)
- **Конфигурация** — Viper
- **Кэширование** — Redis 7 (cache-aside)
- **Метрики** — Prometheus + Grafana
- **Логирование** — Slog + lumberjack (ротация логов)
- **Документация:** OpenAPI 3.0
- **Контейнеризация** — Docker, GitHub Container Registry (GHCR)
- **CI/CD** — GitHub Actions (workflow_dispatch)
- **Архитектура** — Clean Architecture

## Архитектура
Проект построен на принципах **Clean Architecture** с четким разделением на слои:
- **Handler** — HTTP-обработчики (`Gin`)
- **Service** — бизнес-логика приложения
- **Repository** — слой доступа к данным (`PostgreSQL`, `Redis`)
- **Domain** — модели, не зависящие от внешних фреймворков

## Быстрый старт
### Предварительные требования
- Go 1.25+
- Docker и Docker Compose

### Локальный запуск
1. **Клонировать репозиторий**
2. **Настроить конфигурацию**
```bash
    cp config.example.yml config.yml
    # В соответствии с config.example.yaml, создайте файл config.yaml, указав необходимые параметры
```
3. **Запустить инфраструктуру**
```bash
    docker-compose up -d
```
4. **Запустить приложение**
```bash
    go run cmd/main.go
```
Сервер будет доступен на http://localhost:8080
Prometheus будет доступен по адресу: http://localhost:9090
Grafana будет доступен по адресу: http://localhost:3000 (логин/пароль: admin/admin)

## Особенности реализации
### Транзакции и целостность данных
- Все финансовые операции выполняются в рамках PostgreSQL-транзакций (`BEGIN`/`COMMIT`/`ROLLBACK`)
- Блокировка строк `SELECT ... FOR UPDATE` для предотвращения гонок при списаниях
- Атомарное списание и зачисление средств
- Проверка достаточности баланса перед списанием
- Валидация валют: переводы возможны только между счетами в одной валюте
- Запрет на переводы в рамках одного счёта
- При удалении счета обязателен нулевой баланс
- Системный пользователь (нулевой администратор) со счетами для операций пополнения/списания

### Кэширование
Реализовано с использованием `Redis` (стратегия cache-aside):
- **Стратегия:** cache-aside (данные сначала ищутся в `Redis`, при промахе — `PostgreSQL`)
- **TTL:** настраиваемое время жизни записи (по умолчанию 10 минут)
- **Кэшируются:** пользователи по ID, счета по ID пользователя
- **Инвалидация:** автоматическое удаление из кэша при обновлении или удалении данных
- **Graceful degradation:** при недоступности `Redis` сервис продолжает работу напрямую с `PostgreSQL`

### Логирование
- Структурированное логирование с уровнями `DEBUG`, `INFO`, `WARN`, `ERROR`
- Запись логов в файл с автоматической ротацией (`lumberjack`)
- Единый формат сообщений
- Логирование всех ключевых операций: создание пользователя, транзакции, изменение ролей

### Graceful Shutdown
При получении сигнала завершения (`SIGINT`, `SIGTERM`) сервер:
1. Прекращает принимать новые соединения
2. Ожидает завершения активных запросов (таймаут 10 секунд)
3. Корректно закрывает соединение с `PostgreSQL`
4. Корректно закрывает соединение с `Redis`
5. Обработка паник с `recover()` и корректным завершением

### Безопасность
- Пароли хэшируются с помощью **bcrypt** (cost factor = 10)
- **JWT-токены** хранятся в HttpOnly cookie (защита от XSS)
- Cookie с флагами `HttpOnly` и `Secure` (в production)
- Валидация email, номера телефона и временной зоны через регулярные выражения
- Проверка уникальности email и телефона при регистрации и обновлении
- Настроены политики **CORS** для безопасного взаимодействия с фронтендом
- **Role-based middleware** для защиты административных и верификаторских эндпоинтов
- Запрет на выполнение переводов администраторами и верификаторами

## Мониторинг
Сервис экспортирует метрики для Prometheus (`/metrics`):
- `http_requests_total` — счетчик HTTP-запросов (method, endpoint, status)
- `http_request_duration_seconds` — гистограмма длительности запросов
- `http_requests_in_flight` — текущее количество обрабатываемых запросов
- `app_operations_total` — счетчик бизнес-операций (SignUp, CreateAccount, CreateTransaction и др.)
- `app_errors_total` — счетчик ошибок по типам (bad_request, forbidden, internal, business_logic)
- `go_goroutines_count` — количество горутин
- `go_memstats_alloc_bytes` — потребление памяти
- `go_gc_duration_seconds` — длительность пауз сборщика мусора
Готовый дашборд для Grafana: `grafana/dashboard.json`

**Дашборд включает:**
- RPS (Requests Per Second) — скорость запросов по методам и статусам
- Error Rate (4xx + 5xx) — процент ошибочных запросов
- P95 Response Time — 95-й перцентиль времени ответа
- Active Requests — текущее количество обрабатываемых запросов
- Goroutines — мониторинг количества горутин
- Response Time by Endpoint — детализация времени ответа по эндпоинтам
- Business Errors — бизнес-ошибки в реальном времени
- Memory Usage — потребление памяти (Allocated + System)
- GC Pause Time — паузы сборщика мусора

## Контейнеризация
Docker Compose включает:
- **PostgreSQL 17** — основная БД
- **Redis 7** — кэширование
- **Prometheus** — сбор метрик
- **Grafana** — визуализация
Docker-образ приложения основан на `golang:1.25-alpine` и публикуется в GitHub Container Registry.

## CI/CD
GitHub Actions настроен на:
- Автоматическую сборку Docker-образа при ручном запуске (`workflow_dispatch`)
- Публикацию образа в **GitHub Container Registry** (`ghcr.io`)
- Тегирование образа как `latest`
